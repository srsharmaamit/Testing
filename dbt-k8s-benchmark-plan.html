00

What we are measuring & why

Benchmarking dbt on K8s means isolating three distinct layers — the orchestration/pod layer, the dbt process layer, and the warehouse/query engine layer — because a slow run can originate in any of them.

K8Orchestration layer

Pod scheduling latency, image pull time, init-container overhead, CPU/memory throttling, node autoscaling lag, and concurrency limits across the dbt pods.

dbdbt process layer

Parse/compile time, manifest generation, thread concurrency, node selection (state:modified), adapter connection pooling, and run-results timing per model.

SQLWarehouse layer

Query execution time, materialization strategy (table vs incremental vs view), shuffle/scan volumes, partition pruning, and concurrent-query contention.

The benchmark plan — phases & tasks

Click any phase to expand its tasks and step-by-step work. Designed to be executed sequentially; phases 4–6 form the core measurement loop.

P1Define objectives, scope & success criteriaPlanning+
T1.1Establish benchmark goals & SLOs
  1. Interview stakeholders to capture the business question — e.g. "can the nightly run finish in < 90 min at 3× data volume?"
  2. Translate into quantifiable SLOs: total run wall-clock, p95 per-model time, pod startup latency, cost per run.
  3. Classify each metric as latency, throughput, resource, or reliability so dashboards group correctly.
  4. Document explicit pass/fail thresholds that will later drive the regression gate (Phase 8).
T1.2Fix the scope & variables under test
  1. List the independent variables you will sweep: dbt thread count, pod CPU/mem requests, warehouse size, data volume, concurrency.
  2. Pin everything else as controlled constants (dbt version, adapter version, image digest, K8s version) and record them.
  3. Decide the execution model under test: single dbt pod multi-threaded, vs many pods via KubernetesPodOperator / dbt task fan-out.
  4. Write a one-page test charter — anyone re-running it must produce comparable numbers.
P2Build the benchmark harness & environmentSetup+
T2.1Provision an isolated, reproducible cluster
  1. Stand up a dedicated namespace (or cluster) so neighbour workloads don't pollute measurements.
  2. Pin a fixed node pool with known instance types; disable bursting/credits where possible to avoid noisy CPU.
  3. Apply ResourceQuota + LimitRange so pod sizing is deterministic across runs.
  4. Capture the full infra spec as code (Terraform/Helm values) and commit it alongside results.
# pin node pool + quota for the bench namespace apiVersion: v1 kind: ResourceQuota metadata: {name: dbt-bench-quota, namespace: dbt-bench} spec: hard: {requests.cpu: "32", requests.memory: 128Gi, pods: "60"}
T2.2Build a versioned, slim dbt runner image
  1. Build a multi-stage image with pinned dbt-core + adapter versions; record the immutable image digest, not the tag.
  2. Pre-bake the dbt deps (packages.yml) into the image to remove network variance at runtime.
  3. Push to a registry colocated with the cluster to minimise image-pull time; pre-pull onto nodes if needed.
  4. Verify cold-start vs warm-start pull times and note both.
T2.3Wire the orchestration entrypoint
  1. Define how dbt is invoked: a single Job/Pod, or fan-out via KubernetesPodOperator (Airflow) / Argo Workflows.
  2. Externalise profiles.yml via Secret/ConfigMap so warehouse creds and thread counts are injectable per run.
  3. Ensure the runner emits structured logs + run_results.json + manifest.json to a persistent volume or object store.
  4. Add a run-id label to every pod for later correlation in metrics and logs.
P3Generate representative workloads & datasetsData+
T3.1Select / synthesise the dataset
  1. Choose a standard scalable schema (TPC-DS / TPC-H) or a sanitised clone of your production models.
  2. Generate multiple scale factors (e.g. SF=10, 100, 1000) so you can plot scaling curves, not a single point.
  3. Match real-world skew and cardinality — uniform synthetic data hides partition-pruning and join-skew problems.
  4. Stage data into the warehouse once and snapshot it so every run starts from identical state.
T3.2Curate a graduated dbt project
  1. Build model tiers: views, full tables, and incremental models to exercise each materialization path.
  2. Include fan-out and fan-in DAG shapes so dbt's thread scheduler is genuinely stressed.
  3. Add a realistic mix of tests, snapshots, and seeds to reflect a true production run profile.
  4. Tag models so you can run targeted subsets (dbt run --select tag:bench_core).
P4Instrument metrics & observabilityTelemetry+
T4.1Stand up the metrics stack
  1. Deploy Prometheus + node-exporter + kube-state-metrics to capture pod CPU/mem, throttling, and scheduling latency.
  2. Add cAdvisor container metrics so you can see per-pod resource saturation during a run.
  3. Stand up Grafana with a dashboard keyed on the run-id label from T2.3.
  4. Ship dbt logs to Loki / Elasticsearch for searchable per-model timing.
T4.2Capture dbt-native timing artifacts
  1. Parse run_results.json after every run to extract per-node execution_time, status, and thread id.
  2. Diff manifest.json compile time vs total run time to separate parse overhead from execution.
  3. Push parsed timings into a results table (or DuckDB) for trend analysis across runs.
  4. Correlate the slowest dbt models with warehouse query history (query_id ↔ model).
T4.3Instrument the warehouse / query engine
  1. Pull query execution plans & scan bytes (Spark UI / Trino query stats / Snowflake QUERY_HISTORY).
  2. Track queue time vs execution time separately — concurrency contention often hides here.
  3. Record warehouse cost units (credits / DBUs / node-seconds) per run for cost-per-run analysis.
P5Design & run benchmark scenariosExecution+
T5.1Establish the baseline
  1. Run a warm-up pass first and discard it (caches, JIT, connection pools settle).
  2. Execute the baseline configuration N≥5 times to characterise variance, not just the mean.
  3. Record p50 / p95 / p99 + standard deviation for each metric; a noisy baseline invalidates comparisons.
  4. Freeze this baseline as the reference point all subsequent scenarios compare against.
T5.2Sweep the configuration matrix
  1. Vary one dimension at a time: dbt threads (4→8→16→32), then pod CPU/mem, then warehouse size.
  2. Find the thread saturation point — where adding threads stops improving wall-clock or starts throttling pods.
  3. Test fan-out execution (many KubernetesPodOperator tasks) vs single multi-threaded pod for the same DAG.
  4. Capture each cell of the matrix as a labelled run; never change two variables at once.
T5.3Scale & concurrency tests
  1. Re-run the project at each data scale factor (SF10/100/1000) to plot the scaling curve.
  2. Run multiple concurrent dbt invocations to simulate competing teams hitting one warehouse.
  3. Observe whether bottleneck shifts from pod scheduling → dbt threads → warehouse contention as load grows.
P6Load injection, stress & resilienceLoad & Chaos+
T6.1Drive load & find the breaking point
  1. Ramp concurrent runs/queries until a resource (CPU, mem, connections, warehouse slots) saturates.
  2. Use k6 / Locust against the query engine endpoint to layer synthetic query load beneath the dbt run.
  3. Record the knee of the curve — the load level where p95 latency degrades non-linearly.
  4. Confirm whether K8s HPA / cluster-autoscaler reacts in time or lags behind the spike.
T6.2Resilience & failure-mode tests
  1. Kill a dbt pod mid-run (chaos) and verify retry / idempotency of incremental models.
  2. Throttle a node's CPU and confirm the run degrades gracefully, not catastrophically.
  3. Test OOMKill behaviour — does a too-small mem request silently fail or surface a clear error?
  4. Validate state:modified reruns only rebuild the affected sub-DAG after partial failure.
P7Analyse, profile & optimiseAnalysis+
T7.1Identify the bottleneck layer
  1. Overlay pod CPU/mem timelines against the dbt run_results timeline to see where time is actually spent.
  2. Decompose total run time into pod-start + parse/compile + queue + execute and find the largest slice.
  3. Build a critical-path analysis of the DAG — the longest chain bounds your best-case wall-clock.
  4. Flag the top-10 slowest models and tag each as warehouse-bound, thread-starved, or pod-bound.
T7.2Apply & re-measure optimisations
  1. Tune dbt thread count to the saturation point found in T5.2; right-size pod requests/limits.
  2. Convert heavy full-refresh models to incremental; add partitioning / clustering keys in the warehouse.
  3. Reduce pod cold-start via image pre-pull and pod warm-pools where the orchestrator supports it.
  4. Re-run the baseline scenario and quantify the delta — every optimisation must be proven, not assumed.
P8Report, automate & gate regressionsProductionise+
T8.1Produce the benchmark report
  1. Publish results with environment spec, methodology, charts, and percentiles — reproducibility is the deliverable.
  2. Show scaling curves and the config matrix so readers see trade-offs, not just one headline number.
  3. State the recommended production config with its cost-per-run and SLO headroom.
T8.2Automate as a repeatable CI benchmark
  1. Wrap the harness in a scheduled CI job (nightly / per-release) running the fixed baseline scenario.
  2. Store results in a time-series store and chart trend over time to catch slow drift.
  3. Add a regression gate: fail the pipeline if p95 run-time regresses beyond the T1.1 threshold.
  4. Alert on cost-per-run increases as well as latency — both are regressions.

03

Recommended toolchain

MMeasurement & observability

Prometheus + kube-state-metrics + cAdvisor for cluster signals; Grafana dashboards keyed on run-id; Loki/ELK for dbt logs; DuckDB or a results table for parsing run_results.json across runs.

LLoad & chaos injection

k6 or Locust to layer synthetic query load on the warehouse endpoint; LitmusChaos / chaos-mesh for pod-kill, CPU-throttle, and OOM scenarios; concurrent dbt invocations to model multi-team contention.

OOrchestration under test

Airflow KubernetesPodOperator or Argo Workflows for fan-out; a plain K8s Job for the single multi-threaded pod baseline. Externalise profiles.yml via Secret/ConfigMap for per-run tuning.

DData & workload generation

TPC-DS / TPC-H generators at multiple scale factors, or a sanitised production clone with realistic skew. Snapshot the warehouse state so every run starts identical.

Indicative execution timeline

A pragmatic ~5-week cadence for a first full benchmark cycle; compress for a focused single-config test.

Week 1

Phases 1–2 · Charter & harness

Lock objectives/SLOs, provision the isolated cluster, build the pinned runner image, wire the orchestration entrypoint.

Week 2

Phases 3–4 · Data & telemetry

Generate scaled datasets, curate the graduated dbt project, stand up Prometheus/Grafana and dbt artifact parsing.

Week 3

Phase 5 · Baseline & matrix

Establish a low-variance baseline, sweep threads / pod sizing / warehouse size, run scale & concurrency tests.

Week 4

Phases 6–7 · Stress & optimise

Drive load to the breaking point, run resilience/chaos tests, profile bottlenecks, apply and re-measure optimisations.

Week 5

Phase 8 · Report & automate

Publish the reproducible report, wrap the harness as a scheduled CI benchmark with a percentile + cost regression gate.
