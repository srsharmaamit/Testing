Key Insight
HMS only stores a pointer to the current metadata location. It doesn't understand or track the rich snapshot history, branches, or ACID semantics inside Iceberg's metadata layer.

Operational Overhead
HMS is not just one service - it's an entire ecosystem that requires:
  • Database maintenance (backups, scaling, HA)
  • Ranger deployment and configuration
  • Lock coordination service
  • Network latency considerations
  • Single-region deployment constraints

Write Operation Flow with HMS

Step 1: Acquire Lock
Writer requests global lock from HMS
↓ (blocking)
Step 2: Wait for Lock
Other writers blocked until lock released
↓
Step 3: Read Metadata
Fetch current metadata location from HMS
↓
Step 4: Write Data
Write data files + create new metadata
↓
Step 5: Update HMS
Update metadata_location pointer
↓
Step 6: Release Lock
Finally allow other writers to proceed
Problem: Pessimistic Locking
HMS uses pessimistic locking (acquire lock first), while Iceberg is designed for optimistic concurrency (try to commit, fail fast if conflict). This creates a mismatch.

HMS Drawbacks with Iceberg

1. No Snapshot Isolation Understanding
HMS treats each commit as a simple metadata pointer update. It doesn't understand:
  • Snapshot lineage and parent-child relationships
  • Concurrent snapshot creation
  • Snapshot expiration policies
  • Time travel queries across snapshots
-- This works but HMS has no idea SELECT * FROM table FOR SYSTEM_TIME AS OF '2024-01-01 10:00:00'
2. No Multi-Table Transactions
Each table commit is independent. You cannot atomically commit changes across multiple Iceberg tables.
-- NOT POSSIBLE with HMS BEGIN TRANSACTION; UPDATE table_a SET ...; UPDATE table_b SET ...; COMMIT; -- ✗ No atomic commit
This is critical for maintaining consistency across related tables (fact + dimension tables, for example).
3. Global Lock Contention
HMS uses pessimistic locking for table updates. All concurrent writers queue up waiting for the lock.

Impact: Poor write scalability. Instead of Iceberg's optimistic concurrency (try fast, fail fast), you get slow serialized writes.

Iceberg's model: Multiple writers can work in parallel and only conflict if they modify overlapping data.
4. Requires External Governance (Ranger)
HMS has no built-in fine-grained access control. You need Apache Ranger for:
  • Row-level security
  • Column masking
  • Audit logging
  • Policy management
Extra infrastructure: Another service to deploy, monitor, and maintain.
5. Single Region / Latency Issues
HMS is typically deployed in a single region. For multi-region workloads:
  • Every metadata operation crosses regions
  • Added 50-200ms+ per operation
  • No local caching benefit
  • DR/HA becomes complex
Modern catalogs support multi-region deployments natively.
6. No Branch/Tag Support
Iceberg supports branches and tags for:
  • Feature branch development
  • Data quality testing
  • Rollback points
  • Experimentation
HMS has no concept of these - it only tracks the "main" branch pointer.
7. Operational Complexity
Running HMS means maintaining:
  • RDBMS: MySQL/PostgreSQL cluster
  • HMS Service: Java process, memory tuning
  • Ranger: Policy server, audit db
  • ZooKeeper: For locks (optional but common)
  • Monitoring: All of the above
8. Schema Caching Issues
HMS caches schema information, but Iceberg schemas can evolve rapidly. This leads to:
  • Stale schema reads
  • Cache invalidation complexity
  • Inconsistency between HMS and Iceberg metadata
Modern catalogs treat Iceberg metadata as source of truth.

Visual Summary: The Mismatch

HMS Design Philosophy
• Built for Hive tables (2000s)
• Centralized metadata database
• Pessimistic locking
• Schema-centric
• Single table operations
• No versioning concept
• External governance required
Iceberg Design Philosophy
• Built for cloud object stores (2017+)
• Distributed metadata files
• Optimistic concurrency
• Snapshot-centric
• Multi-table transactions
• Git-like versioning
• Catalog-level governance
The Result
HMS acts as a "dumb pointer registry" that doesn't leverage Iceberg's advanced features. You get basic functionality but miss out on the innovations that make Iceberg powerful.

Nessie: Git for Data

Why Nessie is Special
Nessie treats your entire data lake like a Git repository. You can:
Branch Isolation
Create branches for testing, experimentation, or development without affecting production:
-- Create dev branch CREATE BRANCH dev FROM main -- Work on dev USE BRANCH dev UPDATE table SET ...
Atomic Multi-Table Commits
Commit changes across multiple tables atomically:
-- Both succeed or both fail INSERT INTO facts ... INSERT INTO dimensions ... COMMIT
Catalog Time Travel
Query the entire catalog as it existed at any point:
-- See all tables at timestamp SELECT * FROM catalog AT TIMESTAMP '2024-01-01'

When to Use What

Stick with HMS If...
✓ You already have HMS infrastructure
✓ You only need basic Iceberg functionality
✓ Single region deployment
✓ Low write concurrency
✓ No need for branches/tags
✓ Tight integration with legacy Hive workloads
Switch to Modern Catalogs If...
✓ High concurrent write workloads
✓ Need branches/tags for testing
✓ Multi-region deployment
✓ Want multi-table transactions
✓ Need catalog-level governance
✓ Building new lakehouse from scratch
Migration Path
Good news: You don't have to migrate all at once. Modern catalogs can coexist with HMS:
  • Start with new tables in modern catalog
  • Keep existing tables in HMS
  • Gradually migrate high-value tables
  • Many engines support multiple catalogs simultaneously
