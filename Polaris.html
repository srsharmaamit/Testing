What is Apache Polaris?

Apache Polaris is an open-source, fully-featured catalog for Apache Iceberg that implements the Iceberg REST API. Originally created by Snowflake and donated to the Apache Software Foundation, Polaris provides centralized metadata management with enterprise-grade governance built directly into the catalog layer.

üéØ Core Purpose
Centralized catalog for managing Apache Iceberg tables with built-in security and governance
üîê Governance First
RBAC (Role-Based Access Control) is a first-class citizen, not an afterthought
üåê Multi-Engine Support
Works with Spark, Flink, Trino, Dremio, Snowflake, and any Iceberg REST client
üöÄ Key Innovation
Unlike traditional catalogs like HMS that treat governance as an external concern (requiring Apache Ranger), Polaris has governance baked into the catalog itself through its sophisticated RBAC model and credential vending system.

Polaris RBAC Model

Polaris uses a two-layer RBAC model that separates identity management from resource permissions. This design provides maximum flexibility and security.

The Two-Layer Model

Service Principals
(Users/Services/Apps)
‚Üí
Principal Roles
(DataEngineer, DataScientist)
Granted To ‚Üì
Catalog Roles
(TableManager, Reader, Admin)
‚Üí
Resources
(Catalogs, Namespaces, Tables)
Why Two Layers?
Layer 1 (Principal ‚Üí Principal Role): Groups users by job function
Layer 2 (Catalog Role ‚Üí Resources): Defines what actions are allowed on which resources

This separation allows you to manage hundreds of users across multiple catalogs efficiently.

üîí Security Best Practice
Never grant privileges directly to service principals. Always use:

OPA Integration

  • Implement complex, context-aware policies (time-based, IP-based, etc.)
  • Centralize policy management outside the catalog
  • Version control your policies in Git

Credential Vending System

One of Polaris's most powerful security features is credential vending - the ability to issue short-lived, scoped credentials to query engines at runtime.

Traditional Approach (Insecure)
Query engine has permanent AWS keys
Full S3 bucket access
Credentials stored in config files
‚ö†Ô∏è High security risk
vs
Polaris Credential Vending (Secure)
Query engine has NO storage credentials
Polaris vends temporary credentials per-query
Credentials expire after query completes
‚úì Minimal attack surface

How Credential Vending Works

Step 1: Query Submission
User: SELECT * FROM catalog.ns.table
Query engine ‚Üí Polaris REST API
‚Üì
Step 2: Authorization Check
Polaris validates:
‚Ä¢ Service principal identity
‚Ä¢ Principal roles
‚Ä¢ Catalog roles
‚Ä¢ TABLE_READ_DATA privilege
‚Üì
Step 3: Credential Vending
Polaris generates temporary credentials:
‚Ä¢ STS AssumeRole (AWS)
‚Ä¢ Service Principal (Azure)
‚Ä¢ Workload Identity (GCP)

Credentials scoped to ONLY this table's S3 prefix
Expiration: 1 hour (configurable)
‚Üì
Step 4: Data Access
Query engine uses vended credentials to:
‚Ä¢ Read metadata.json from S3
‚Ä¢ Read manifest files
‚Ä¢ Read data files (Parquet)

Credentials automatically expire after query
Security Benefits
‚úì No Long-Lived Credentials: Query engines never store permanent cloud credentials
‚úì Least Privilege: Credentials are scoped to exactly what's needed for the query
‚úì Automatic Expiration: Even if stolen, credentials expire quickly
‚úì Centralized Management: All storage access flows through Polaris
‚úì Audit Trail: Every credential issuance is logged
