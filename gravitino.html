Apache Gravitino

The Unified Metadata Lake - Catalog of Catalogs

What is Apache Gravitino?

Apache Gravitino is a high-performance, geo-distributed, and federated metadata lake. Unlike traditional catalogs that manage a single metadata source, Gravitino is a "Catalog of Catalogs" - a unified metadata management platform that provides governance and access across heterogeneous data sources, regions, and clouds.

üåê Unified Metadata
Manage diverse metadata through a single model and API: Hive, Iceberg, MySQL, PostgreSQL, HDFS, S3, Kafka, ML models
üîó Federation First
Connect to existing catalogs (Polaris, Nessie, Unity) without replacing them - add governance on top
üåç Geo-Distribution
Deploy across regions and clouds, share metadata globally while maintaining locality
üöÄ Key Innovation: The Metadata Lake
Gravitino introduces the concept of a Metalake (metadata lake) - a tenant-like container that manages ALL your metadata across different sources, types, and regions. Think of it as a data brain: a structured, queryable, and semantically enriched representation of all your data assets.

The Gravitino Philosophy

Unified, Not Uniform
Gravitino doesn't force everything into one format. It provides unified APIs while respecting the native capabilities of each source system.
Federation Over Migration
Don't replace your existing catalogs - connect to them! Gravitino federates metadata from Hive Metastore, JDBC databases, cloud-native solutions.
Direct Integration
Changes in Gravitino reflect directly in underlying systems, and vice versa. No passive metadata collection - active management.
Multi-Modality Support
Not just tables! Gravitino manages relational data, files, Kafka topics, and ML models through consistent interfaces.

Gravitino RBAC Model

Gravitino implements a hierarchical RBAC system similar to traditional database security models, but applied across the entire metadata lake.

1. Users
Identity Principals:

‚Ä¢ Added to metalakes explicitly
‚Ä¢ Can be individuals or service accounts
‚Ä¢ Each user can have multiple roles

Important: Users must be added to metalake before they can do anything!
2. Roles
Grouping Mechanism:

‚Ä¢ Created within a metalake
‚Ä¢ Assigned privileges on securable objects
‚Ä¢ Granted to users

Example Roles:
‚Ä¢ catalog_admin
‚Ä¢ data_engineer
‚Ä¢ analyst_reader
3. Privileges
Specific Actions:

Metalake: CREATE_CATALOG, MANAGE_USERS
Catalog: USE_CATALOG, CREATE_SCHEMA
Schema: USE_SCHEMA, CREATE_TABLE
Table: SELECT_TABLE, MODIFY_TABLE
Fileset: READ_FILESET, WRITE_FILESET
Hierarchical Inheritance
Key Feature: Privileges are inherited downward in the hierarchy:

‚Ä¢ Grant USE_CATALOG on a catalog ‚Üí Automatically applies to all schemas and tables within
‚Ä¢ Grant CREATE_TABLE on a schema ‚Üí User can create tables in that schema
‚Ä¢ More specific grants override general ones

RBAC in Action

# Scenario: Data Engineering Team Setup # 1. Add Users to Metalake POST /api/metalakes/prod/users { "name": "alice@company.com" } POST /api/metalakes/prod/users { "name": "bob@company.com" } # 2. Advanced: Push-Down Permission Management

Gravitino's "Push-Down Permission Management" means that RBAC policies can be enforced at the source system level, not just in Gravitino.

How It Works
When you grant privileges in Gravitino, it can optionally:

1. Store in Gravitino: Enforce when accessing via Gravitino API
2. Push to Source: Create corresponding permissions in Hive/MySQL/etc.

This means users accessing data directly (bypassing Gravitino) still respect permissions!
üí° Enforcement Modes (v1.0.0+)
Prior to v1.0.0, Gravitino could only define permissions.
From v1.0.0 onwards, Gravitino actively enforces access control when manipulating secure objects.

Enforcement happens when:
‚Ä¢ Users create/drop tables via Gravitino API
‚Ä¢ Query engines access metadata through Gravitino IRC
‚Ä¢ Applications use Gravitino connectors (Spark, Trino, Flink)
